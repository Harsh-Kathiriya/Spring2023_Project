Short Paragraph


Our team searched the Chocoholics Anonymous project document and identified nine use cases involving the Chocoholics Anonymous data center. For each use case, we identified the actors involved and discussed its general success scenario. We then divided the use cases amongst five team members and each member individually fleshed out his or her cases and later added them to this Report.html file. After creating the use cases, we identified all of the technical terms in our use case descriptions that were defined in the ChocAn project document and added them to our glossary alongside their definitions from the document. Then, each member created a use case diagram depicting the use cases we identified as a team. We decided on a diagram that best illustrated our use cases and included it in this Report.html file. Each individual use case diagram is included as a png file in our requirements folder and attached to our individual reports, the reports each containing two tables showing each member's student information and task distribution respectively. We will be coding these requirements in Java.


Glossary


  1. Accounting Process - Weekly process of reading all ChocAn services provided in that week and printing reports of each one.


  2. Acme - Third-party organization responsible for financial procedures such as recording payments of membership fees, suspending members whose fees are overdue, and reinstating suspended members who have now paid what is owed.


  3. ChocAn data center- Data center stores all data related to ChocAn services and is accessed to create reports of a variety of ChocAn data.


  4. Chocoholics Anonymous - An organization dedicated to helping people addicted to chocolate. Members pay a monthly fee to ChocAn in exchange for access to addiction help services.


  5. EFT Record - Record created by the data center of funds to be transferred from ChocAn to its various providers.


  6. Manager - ChocAn employee that can view ChocAn data reports and verifies fees to be paid to ChocAn providers


  7. Member Record - Record of all information associated with an individual member using ChocAn’s services. Can be updated during the day by operators.


  8. Member Report - Record for individual members that includes the member name, number, street address, city, state, and zip code. For each service provided, it includes the date of service, provider name, and service name.


  9. Operator - A person that interacts with the ChocAn data center to add, delete, or update member records


  10. Provider - Health care professional that provides services to ChocAn members


  11. Provider directory- A directory that is accessed when providers want to verify that the correct service was provided. This includes the current date and time, the date service was provided, provider number, member number, service code, and comments.


  12. Provider Report - A report was given to each provider at the end of the week detailing all the services he or she has provided to ChocAn members during that week.


  13. Service Record - A record of a service provided, created by the data center to be sent to ChocAn for billing. Includes date, provider and member number, and a code corresponding to the type of service provided.


  14. Summary Report - A report given to the manager lists every provider to be paid that week, the number of consultations each had, and his or her total fee for that week.


  15. Terminal - Device that the provider interacts with to communicate with the ChocAn Data Center


  16. Timer - An automatic timer that activates at midnight on Friday to run the Main Accounting Procedure

Main Use Case 1:

Use Case: Validate Member

Context: The provider uses a terminal in order to gain access to member services.

Actors: Provider

Main Success Scenario:

  1. The provider scans the member card through the card reader.

  2. The member’s card number is sent to Chocoholics Anonymous Data Center through the terminal.

  3. Data Centers then verify the card number.

  4. Upon successful verification, “Validated” is displayed on one line display.

  5. The Validate Member process terminates and returns to the provider menu.

Extensions:

  2a. The card number is not valid.

    2a.1 “Invalid Number” is displayed on one line display.

    2a.2. The use case continues from step 1.

  2b. The member is suspended because of dues.

   2b.1 “Member Suspended” is displayed on one line display.

   2b.2. The system continues from step 1.

Use Case: Bill Chocoholics Anonymous

Context: After a health care service has been provided, the provider intends to send a bill to Chocoholics Anonymous.

Actors: Provider

Main Success Scenario:

  1. The provider swipes the card through the card reader or types in the member's number.

  2. The word “Validated” will prompt on the screen.

  3. The system then asks the provider to type in the service date.

   3.1. The system displays a text box for the provider to enter the information.

   3.2. It will display in the MM-DD-YYYY format.

   3.3. The provider enters the service date

  4. The provider will then use the provider directory to look up the six-digit service code

  corresponding to the provided service.

   4.1. The provider types in the service code.

  5. The software displays the name of the service and the corresponding code for the provider

  to verify.

   5.1. The provider verifies the correct service and the name has been inputted.

  6. The software then compiles a file containing the following information:

   6.1. Current date and time (MM-DD-YYYY HH:MM:SS).

   6.2. Date service was provided (MM-DD-YYYY).

   6.3. Provider number (9 digits).

   6.4. Member number (9 digits).

   6.5. Service code (6 digits).

   6.6 Comments (100 characters) (optional).

  7. The software then looks up the fee to be paid for that service and the system displays it for the

  provider.

  8. The system displays a form with empty text boxes with labels from step 7 for verification.

   8.1. The provider fills the text boxes with the information

  9. At the end of the week, the provider totals all of the fees to verify the amount to be paid to that

  provider by Chocoholics Anonymous.

  10. Bill Chocoholic Anonymous process terminates and retuns to the provider menu.

Extensions:

  4a. The provider has entered a nonexistent code.

    4a.1. An error message will display informing the provider.

   4a.2. The use case continues from step 5.

  5a. The provider can enter comments about the service

   5a.1. The provider is given a text box to enter any comments they wish.

   5a.2. The use case continues from step 6.

Use Case: Request Summary Report

Context: An actor requests a report for accounts payable.

Actors: Manager, Timer

Main Success Scenario:

  1. The actor requests a summary report.

  2. The system goes through the Chocoholic Anonymous Data Center database to pull information.

   2.1. The system retrieves all the providers that are to be paid that week.

   2.2. The system retrieves the number of consultations from members that week.

   2.3. The system retrieves the total number of providers who provided services.

   2.4. The system retrieves the total number of consultations.

   2.5. The system retrieves the total overall fee for that week.

  1. The system creates a file entailing all of the data retrieved in step 3.

   3.1. A chart is created in the file.

   3.2. The first column has the information in step 2.1.

   3.3. The second column has the information from step 2.2.

   3.4. The third column has the information from step 2.3.

   3.5 The fourth column has the information from step 2.4.

   3.6 The fifth column has the information from step 2.5

  1. The system prints the file.

  2. Request Summary Report process terminates.

Extensions:

  1a. There is an error pulling the request

   1a.1. The system informs the manager.

   1a.2. The use case continues at step 1.

  2a. There is an error in retrieving information

   2a.1. The system informs the manager.

   2a.2. The use case continues at step 1.

  3a. The system cannot create a summary report file

   3a.1. The system alerts the manager that the summary file cannot be created.

   3a.2. The use case continues at step 1.

  4a. The system cannot print out the file.

   4a.1. The system saves all of the information.

   4a.2. The system alerts the manager that there is an error in printing the file.

   4a.3. The use case continues at step 1.
